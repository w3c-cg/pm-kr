This report explains why the PM-KR Community Group should prepare Procedural Memory Knowledge Representation for transition to the W3C standards track, identifies the normative subset that is mature enough to incubate toward a Working Group deliverable, and positions Knowledge3D as the current reference implementation and evidence base.

The report is based on the complete Knowledge3D architecture vocabulary corpus, not on a single summary document. It treats the K3D codebase and vocabulary as the current implementation and design record from which a standards-track PM-KR profile can be extracted.

This is a Draft Community Group Report prepared by the PM-KR Community Group. It is not a W3C Standard and does not imply W3C endorsement. Its purpose is to support Community Group incubation, convergence, and later transition to a standards-track venue.

Introduction

PM-KR is a proposal for a procedural, reference-preserving knowledge representation in which canonical knowledge can be stored once, linked compositionally across layers, and consumed consistently by both humans and machines. The current work is incubated in the PM-KR Community Group and implemented in the Knowledge3D project.

Community Group reports are not standards-track documents, but they can become input to the Recommendation Track. W3C’s process and Community Group guidance make that transition easier when the group builds clear reports, maintains continuity of IPR commitments, and develops work that can be adopted by an existing or newly chartered Working Group.

This document is therefore not an explainer for the K3D project. It is a W3C-facing transition report: what PM-KR is trying to standardize, which parts are grounded enough to move forward, what remains reference implementation detail, and what evidence the group should gather before a standards-track handoff.

User-Facing Problem

Knowledge that needs to be shared between humans, software, and AI systems is usually fragmented across incompatible representations. The same concept may appear as prose in a document, duplicated entries in a graph or database, runtime-only program state, and hidden model-internal parameters. This fragmentation makes knowledge harder to verify, less portable, and more expensive to maintain.

The web standards ecosystem already knows how to standardize interoperable documents, linked data, graphics, and APIs. PM-KR targets a different but adjacent need: a portable way to preserve canonical procedural knowledge, explicit references between layers of meaning, and dual-client equivalence between human-facing and machine-facing views of the same node.

Goals

  • Define a canonical knowledge object model that distinguishes form, meaning, executable rules, and higher-order control.
  • Preserve references to canonical lower layers instead of duplicating payloads across authoring, storage, and execution systems.
  • Support deterministic reconstruction and auditability for a fixed implementation profile.
  • Preserve equivalence across human and synthetic-user views of the same canonical node.
  • Define an interoperability target that is independent from any single runtime engine while still benefiting from a working reference implementation.

Non-goals

  • This report does not propose to standardize a specific GPU backend, a specific sovereign kernel runtime, or a specific neural model.
  • This report does not attempt to move the entire K3D architecture to the standards track as one monolithic deliverable.
  • This report does not claim that Community Group work itself is a W3C Standard.

Use Cases

  • A standards-oriented knowledge system stores a canonical concept once and reuses it through stable references from executable rules and UI surfaces.
  • A human reviewer and a synthetic user inspect the same knowledge node through different presentation modes without losing identity or provenance.
  • A reference implementation ingests external material into a canonical node model while preserving lineage and reconstruction metadata.
  • A second implementation can reconstruct or exchange the same node graph without depending on the first implementation’s internal hot path.
  • Accessibility and localization layers can reuse canonical semantic nodes without requiring separate authoritative payloads.

Success Criteria

A future standards-track PM-KR effort should be considered successful only if it demonstrates real interoperability value beyond the current reference implementation.

  • At least two independent implementations can exchange or reconstruct the same PM-KR core node model with consistent semantics.
  • The standards-track subset is small enough to be reviewed and adopted independently from K3D-specific runtime architecture.
  • Conformance tests cover canonicality, reference preservation, reconstruction, provenance, and dual-client equivalence.
  • Reviewers can trace clear user and implementer value, rather than a request to bless an existing project wholesale.
  • The group can point to broad expressions of interest, implementation feedback, and an adoption path through an appropriate Working Group.

Why This Belongs on a Standards Path

PM-KR addresses an interoperability problem, not just an application architecture preference. The problem is how to describe, preserve, and exchange canonical procedural knowledge in a way that can survive translation across tools, clients, and runtimes.

W3C’s standards-track guidance is centered on mature multi-party deliverables with clear scope and reviewability. W3C’s Community Group process, in turn, is explicitly designed to make it easier for a group’s report to move toward standards-track work by maintaining continuity of IPR commitments, allowing adoption by a relevant Working Group when scope aligns, and turning Community Group Reports into structured inputs for that later step.

PM-KR is therefore best understood as a candidate standards-track topic emerging from Community Group incubation. The standards-track target is not “the whole K3D system,” but a stable interoperability layer extracted from K3D’s full architecture and validated by implementation experience.

Reference Implementation: Knowledge3D

Knowledge3D is the current reference implementation for PM-KR. It should be treated as the implementation and experimentation environment from which the standards-track subset is derived. The PM-KR Community Group should not describe K3D as the future standard itself; instead, K3D provides the architecture corpus, executable evidence, and conformance surface that help the group decide what is mature enough to standardize.

The implementation value of K3D is that it does not stop at abstract terminology. It includes canonical node definitions, runtime boundaries, ingestion pathways, spatial interfaces, layered knowledge contracts, and a living vocabulary used to coordinate design and implementation.

Proposed Standards-Track Scope

A standards-track PM-KR deliverable should initially focus on the parts of the architecture that define portable knowledge meaning and exchange.

Candidate normative core

  • The four-layer model: Form, Meaning, Rules, and Meta-Rules.
  • Canonicality and reference-preservation invariants.
  • Minimal node contract, stable identifiers, and conformance profiles.
  • Dual-client equivalence requirements for shared node identity.
  • Deterministic reconstruction and provenance expectations.
  • The sovereign boundary between offline ingestion and runtime use as an architectural contract, without standardizing one runtime engine.

Candidate informative material

  • K3D-specific runtime layouts, kernel inventories, and hot-path implementation strategies.
  • Spatial UX patterns such as the Memory Tablet and House/Galaxy environment design.
  • Long-range paradigm framing such as SGI and SHGI.
  • Domain-specific specialist content, benchmark narratives, and project-specific deployment choices.

Transition Path from Community Group to Standards Track

The relevant W3C process guidance is straightforward: Community Group Reports can feed the standards process, but Community Groups do not themselves publish standards-track specifications. To move PM-KR forward, the group should use draft and final Community Group reports to stabilize scope, capture commitments, and accumulate implementation and review evidence for eventual Working Group adoption.

  1. Continue publishing PM-KR Community Group Reports that satisfy the Community Group report requirements.
  2. Advance the PM-KR normative subset to a well-scoped Final Report when the group is ready to invite Final Specification Agreement commitments.
  3. Build implementation evidence beyond one codebase, including at least one non-K3D-oriented or independently developed implementation profile.
  4. Socialize the work with likely standards-track stakeholders and identify whether an existing Working Group could adopt it or whether a dedicated charter would be needed.
  5. Bring the mature normative subset, use cases, tests, and evidence into a Working Group venue for standards-track work.

Socialization and Evidence Gaps

The current PM-KR work has a strong reference implementation and a broad architecture corpus, but it still needs the kind of external evidence that W3C readiness guidance expects for formal standards work.

  • The problem statement is clear, but it should be socialized more widely with potential implementers outside the K3D project.
  • Success criteria are now explicit in this report, but they still need measurable implementation evidence.
  • The proposal is technically mature in K3D terms, but the standards-track subset must be narrowed and shown to be adoptable by other systems.
  • The Community Group should gather public expressions of interest from prospective implementers, tool builders, and reviewers.
  • The group should build an evidence package that can accompany any later Working Group adoption proposal: use cases, tests, implementation reports, opposition or concerns, and IPR readiness.

Readiness Criteria Before Working Group Adoption

  • Clear extraction of the PM-KR normative subset from the larger K3D architecture.
  • Stable terminology and consistent identifier conventions across the complete vocabulary corpus.
  • Conformance and reconstruction tests that are meaningful outside the K3D hot path.
  • Evidence that at least one other implementation or implementation strategy can consume the same core model.
  • Privacy, security, accessibility, and internationalization review of the exchange model.
  • Community Group publishing discipline: draft reports, final reports, and clear contributor and commitment hygiene.

Accessibility, Internationalization, Privacy, and Security Considerations

Accessibility Considerations

PM-KR should preserve one canonical node truth across presentation modalities so that visual, textual, auditory, and assistive representations do not diverge semantically. This is a strength of the proposal, but it must be tested with concrete accessibility mappings and not assumed by architecture alone.

Internationalization Considerations

The separation of form and meaning is favorable for localization and multi-script systems, because language-specific surface forms can be attached to shared canonical concepts. The group should validate this with concrete multilingual exchange examples before standards-track advancement.

Privacy Considerations

Provenance, lineage, and reconstruction metadata are useful for audit, but implementations need clear rules for when source traces may expose sensitive material. The standards-track subset should make room for privacy-preserving profiles and redaction behavior.

Security Considerations

PM-KR deliberately distinguishes offline ingestion from runtime use. That boundary is important for security. The standards-track subset should define safe interchange and validation expectations without assuming trust in arbitrary executable payloads.
